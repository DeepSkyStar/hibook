# 知识管理规则 (Knowledge Management Rules)

> **核心目标**: 知识库不仅是存放数据的仓库，而是经过**理解、拆解、融合**后的结构化思维导图。

## 1. 知识准入标准 (Admission Criteria)
任何试图进入本知识库的内容，必须满足：
1. **去个人化 (Depersonalized)**: 移除主观叙述，提炼为客观的定义、原理或方法论。
2. **结构化 (Structured)**: 必须有清晰的层级（一级标题为核心概念，二级标题为属性/原理，三级标题为应用/案例）。
3. **唯一性 (Uniqueness)**: 检查库中是否已有同类概念。
   * **无重复**: 新建文件，并在 `SUMMARY.md` 中注册。
   * **有重复**: 执行**融合 (Merge)** 操作。
4. **严谨树形 (Strict Hierarchy)**: 知识体系必须是严格的树形结构，内部无断层。

## 2. 目录与组织规则 (Strict Organization Rules)
本知识库基于 `hibook` 驱动，所有的文件组织必须**极其严谨**地遵循以下标准：

1. **结构映射与标题规范**:
   * 所有 Markdown 文档的**第一行必须包含一个 `# 一级标题`**。底层工具 `hibook update` 解析目录树时，会提取这个 H1 作为 `SUMMARY.md` 中的书签显示名称。
   * 强烈建议为每个主题目录提供一个 `README.md`，作为该层级的向导页。
   * 文件列表默认按字母表顺序编译与展示。如果对展示顺序有强烈要求，**建议**在文件或文件夹名前加上数字序列（如 `01-`, `02-`）以控制排序逻辑。
2. **SUMMARY 骨架系统**: `SUMMARY.md` 是整个系统的权威骨架。
   * 严禁越过 `hibook update` 或者手动登记等方式私自添加无索引节点的游离态 `.md` 文件。任何未在 `SUMMARY.md` 中体现的知识块都会被视为“非法结构”，不会被编译出站。
   * 可以使用 `hibook update` 自动基于层级规则与首行 `H1` 标题刷新索引树。
3. **相对路径引用**:
   * 所有文档之间的超链接，必须使用**相对路径**（例如 `../folder/file.md` 或 `file.md`）。
   * 严禁使用绝对硬编码路径（如 `/Users/...` 或 `/data/...`）。
4. **媒体与资源**:
   * 图片或附件应当存放在同级的 `assets/` 或专门的资源文件夹内，并通过相对路径引用。
5. **可视化互动 (Mermaid)**:
   * 鼓励在 `README.md` 或核心文件中使用 `mermaid` 绘制树形图（flowchartTD 等）。
   * 所有的架构图流向都必须附加可点击的路由链接，图形内的交互节点必须使用 `click 节点名 "相对路径"` 绑定，这将被底层库无缝解析。

## 3. 融合操作流程 (The Merge Protocol)
当新知识与现有知识重叠时，严禁直接追加到文件末尾。必须遵循以下步骤：

### 第一步：概念对齐 (Concept Alignment)
* 阅读现有文档，理解其核心定义的边界。
* 阅读新内容，提取其核心定义。
* **判断**: 新旧定义是互补的？矛盾的？还是包含关系？

### 第二步：重构 (Refactoring)
根据对齐结果进行重写：
* **互补**: 将新内容作为现有文档的一个新章节（Section），或者扩充现有章节的细节。
* **矛盾**: 在文档中设立“争议”或“不同视角”板块，对比两种观点。
* **包含**: 如果新内容是旧内容的子集，仅保留独特细节；如果新内容更全面，用新内容替换旧内容。

### 第三步：原子化 (Atomization)
* 如果一篇新文章包含多个领域的知识，必须将其**拆解**并分别归入对应领域的目录下。
* 在相关文档间建立双向的交叉引用。

## 4. 引用与链接规范 (Linking & Referencing)
为了保证知识库的连通性与封闭性，必须遵守以下规则：
* **强制链接 (Mandatory Hyperlinks)**: 凡是在文本中提及库内其他文档模块时，**必须**使用 Markdown 链接格式。
* **知识闭环**: 知识库内的文档只能引用库内（由 `SUMMARY.md` 管辖的体系）的其他文档，确保电子书导出时不存在失效的外部越界跳转。
